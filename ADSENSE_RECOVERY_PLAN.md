# KLEVX AdSense Approval Recovery Plan

## 목적

이 문서는 klevx.com의 Google AdSense 반려 사유인 "가치가 별로 없는 콘텐츠" 문제를 해결하기 위한 제품/콘텐츠/기술 설계안이다.

핵심 방향은 AdSense 승인을 위한 표면적인 페이지 추가가 아니다. KLEVX를 "매수, 전세, 월세를 고민하는 실거주자를 위한 신뢰 가능한 주거비 의사결정 콘텐츠 사이트 + 계산기"로 재구성한다.

## 정책 해석

Google Publisher Policies는 게시자 콘텐츠가 없거나 저가치 콘텐츠인 화면, 공사 중인 화면, 알림/내비게이션 목적의 화면에 Google 광고를 게재하는 것을 허용하지 않는다. 또한 개인정보처리방침에는 Google 제품 사용으로 발생하는 쿠키, 웹 비콘, IP 주소, 제3자 데이터 사용을 명확히 고지해야 한다.

AdSense Program Policies도 광고가 비콘텐츠 기반 페이지나 광고 표시만을 목적으로 만든 페이지에 배치되면 안 된다고 설명한다.

참고:

- Google Publisher Policies: https://support.google.com/adsense/answer/10502938
- AdSense Program Policies: https://support.google.com/adsense/answer/48182

## 현재 문제 진단

현재 앱은 계산 로직과 UI는 갖추고 있지만, 심사 봇 관점에서는 다음 문제가 있을 수 있다.

1. 계산기 중심 구조라 입력 전 페이지가 충분한 게시자 콘텐츠로 보이지 않을 수 있다.
2. 홈이 계산기 진입 CTA 중심이라 독립적인 콘텐츠 사이트의 첫인상이 약하다.
3. `/guide`, `/faq`, `/terms`, `/about`이 있더라도 페이지별 깊이, 출처, 작성 기준, 업데이트 정보가 부족하면 얇은 콘텐츠로 판단될 수 있다.
4. 부동산, 대출, 세금, 투자수익률은 신뢰성이 중요한 영역이므로 출처와 한계 고지가 약하면 승인 리스크가 커진다.
5. 계산 결과는 사용자 입력 후 생성되므로, 크롤러가 계산 결과를 보지 못해도 사이트 가치가 드러나는 정적 콘텐츠가 필요하다.

## 제품 포지셔닝

### 대상 독자

핵심 독자는 투자자가 아니라 실거주 의사결정자다.

예상 독자:

- 집을 살지 전세로 갈지 고민하는 20~40대
- 대출 이자와 월 지출을 감당할 수 있는지 알고 싶은 사용자
- 전세 보증금과 월세 중 어느 쪽이 유리한지 비교하려는 사용자
- 집값 상승률보다 현금흐름과 보유 기간을 먼저 확인하고 싶은 사용자

### 톤

금지:

- "무조건 매수가 유리합니다"
- "수익을 보장합니다"
- "이 선택이 정답입니다"
- 투자 조언처럼 보이는 단정적 표현

권장:

- "이 조건에서는 월 지출 부담이 커질 수 있습니다"
- "보유 기간이 짧으면 초기 비용 회수 가능성이 낮아집니다"
- "계산에 반영되지 않은 항목을 함께 확인해야 합니다"
- "개인 조건에 따라 결과가 달라질 수 있습니다"

## 홈 개편 방향

홈은 계산기 진입 페이지가 아니라 콘텐츠 허브형 첫 화면이어야 한다.

필수 섹션:

1. 서비스 한 줄 정의
   - "매수, 전세, 월세를 초기자금, 월 지출, 순자산 관점에서 비교하는 주거비 계산기"
2. 계산기 CTA
   - 첫 화면에 유지하되 페이지 전체가 CTA만으로 끝나면 안 된다.
3. 대표 가이드 4개
   - 초기자금/월 지출/순자산
   - 기회비용
   - 대출이자
   - 보유 기간
4. 대표 사례 3개
   - 전세가율 70%
   - 대출금리 4.5%
   - 초기자금 1억
5. 계산 방식 요약
   - 반영 항목과 미반영 항목을 짧게 노출
6. 신뢰 링크
   - 방법론, 데이터 출처, 편집 정책, 개인정보처리방침
7. 최근 업데이트 기준일
   - "계산 기준 최종 확인일: YYYY-MM-DD"

## 신뢰 페이지 요구사항

재심사 전 최소 8개의 신뢰 페이지를 만든다. 각 페이지는 독립 URL, 충분한 본문, 내부 링크, 작성/업데이트 정보, 면책 고지를 가져야 한다.

| URL | 목적 | 핵심 내용 |
| --- | --- | --- |
| `/about` | 게시자 정체성 | 서비스 목적, 대상 독자, 계산기가 해결하려는 문제 |
| `/contact` | 연락 가능성 | 이메일 또는 문의 채널, 오류 제보 안내 |
| `/privacy` | 개인정보 고지 | 쿠키, Google 광고, 제3자 광고, IP/식별자, 사용자 동의 |
| `/terms` | 이용 조건 | 계산 결과의 참고용 성격, 책임 제한, 금지 행위 |
| `/editorial-policy` | 편집 기준 | 출처 선정 원칙, 업데이트 주기, 오류 수정 정책 |
| `/methodology` | 계산 방법 | 매수/전세/월세 계산식, 반영 항목, 계산 한계 |
| `/data-sources` | 출처 관리 | 항목별 기준값, 출처, 최종 확인일, 변경 가능성 |
| `/disclaimer` | 면책 고지 | 법률, 세무, 금융 자문이 아니라는 점 |

## 콘텐츠 최소 기준

페이지 수보다 중요한 것은 심사 가능한 완성도다. 재심사 전 콘텐츠는 다음 조건을 충족해야 한다.

- 독립 URL로 접근 가능
- 본문 1,200~2,000자 이상
- 검색 의도가 명확한 제목
- 작성일, 최근 업데이트일, 계산 기준일 표시
- 참고 출처 표시
- 관련 계산기 CTA 포함
- 관련 가이드/사례 내부 링크 포함
- 페이지 하단에 참고용 고지 포함
- "준비 중", placeholder, 빈 섹션 없음

## 사례형 콘텐츠 7개

사례 글은 계산 결과를 복사하는 글이 아니어야 한다. 각 글은 "이 상황에서 사용자가 어떤 판단을 해야 하는가"를 설명해야 한다.

1. `/case-studies/jeonse-ratio-70`
   - 제목: 전세 보증금이 매매가의 70%일 때, 사는 게 나을까?
   - 핵심: 전세가율, 자기자본, 대출 규모, 보유 기간의 관계

2. `/case-studies/loan-rate-4-5`
   - 제목: 대출금리 4.5%에서 6억 아파트 매수와 전세 비교
   - 핵심: 금리 상승이 월 지출과 순자산에 미치는 영향

3. `/case-studies/monthly-rent-150`
   - 제목: 월세 150만원과 전세대출 이자, 실제 월 지출 비교
   - 핵심: 월세와 전세대출 이자의 현금흐름 차이

4. `/case-studies/move-within-5-years`
   - 제목: 5년 안에 이사할 가능성이 높다면 매수는 불리할까?
   - 핵심: 취득세, 중개보수, 보유 기간, 거래비용 회수 가능성

5. `/case-studies/cash-100m`
   - 제목: 초기자금 1억으로 가능한 선택지: 매수, 전세, 월세
   - 핵심: 초기 필요 자금과 월 지출 감당 가능성

6. `/case-studies/flat-or-falling-price`
   - 제목: 집값이 오르지 않거나 하락하면 전세와 월세 중 무엇이 나을까?
   - 핵심: 가격 상승률 0% 또는 하락 시나리오에서의 선택

7. `/case-studies/jeonse-loan-limit`
   - 제목: 전세대출 한도가 부족할 때 비교해야 할 선택지
   - 핵심: 보증금 부족, 월세 전환, 초기자금 부족 리스크

## 가이드형 콘텐츠 8개

가이드 글은 계산 항목 설명보다 판단 프레임을 중심으로 쓴다.

1. `/guide/cashflow-net-worth`
   - 매수, 전세, 월세 비교에서 봐야 할 3가지: 초기자금, 월 지출, 순자산

2. `/guide/opportunity-cost`
   - 전세 보증금의 기회비용은 어떻게 계산할까

3. `/guide/loan-interest`
   - 주택담보대출 이자가 주거 선택에 미치는 영향

4. `/guide/acquisition-tax-agent-fee`
   - 취득세와 중개보수는 매수 판단에서 얼마나 중요할까

5. `/guide/monthly-rent-conversion-rate`
   - 월세 전환율이 높을 때 전세가 유리해지는 이유

6. `/guide/holding-period`
   - 보유 기간이 짧을수록 매수가 불리해지는 이유

7. `/guide/variables-not-in-calculator`
   - 계산기로 알 수 없는 부동산 선택 변수들

8. `/guide/how-to-use-calculator`
   - KLEVX 계산기를 해석하는 방법

## 계산기 신뢰도 보강

결과 화면에는 다음 정보를 고정 노출한다.

### 반영된 항목

- 매수: 취득세, 중개보수, 법무비 추정, 채권할인 추정, 재산세, 종합부동산세, 대출이자, 양도 비용, 시세 변화, 자기자본 기회비용
- 전세: 중개보수, 보증보험료, 전세대출 이자, 보증금 기회비용, 보증금 상승률
- 월세: 보증금 기회비용, 월세 누적, 월세 상승률, 중개보수

### 반영되지 않은 항목

- 관리비
- 수선비
- 이사비
- 인테리어 비용
- 지역별 대출 규제 차이
- 개인 소득 조건
- 청약 기회비용
- 세대별 세무 특례
- 실제 매매 가능 가격과 협상가

## 광고 적용 정책

### 재심사 전

권장 순서:

1. 콘텐츠 작성
2. 신뢰 페이지 완성
3. 기술 SEO 정리
4. Search Console 색인 확인
5. AdSense 심사용 코드 삽입
6. 재심사 요청

콘텐츠와 정책 페이지가 완성되기 전에는 전역 AdSense script를 제거하거나 비활성화하는 편이 안전하다.

### 승인 후

허용 위치:

- 가이드 본문 중간
- 가이드 본문 하단
- 사례 본문 하단
- FAQ 본문 하단

금지 위치:

- 계산 전 빈 상태
- 로딩 화면
- 계산 버튼 주변
- 내비게이션 주변
- footer 정책 링크 주변
- 본문보다 광고가 먼저 보이는 위치
- 광고가 메뉴, 버튼, 다운로드 링크처럼 보이는 위치

## 실행 로드맵

### 1단계: 사이트 정체성 정리

- 홈을 콘텐츠 허브형으로 개편한다.
- 실거주 의사결정자 대상 문구로 통일한다.
- 투자 수익, 고수익, 확정 판단처럼 보이는 표현을 제거한다.

### 2단계: 신뢰 페이지 구축

- `/about`
- `/contact`
- `/privacy`
- `/terms`
- `/editorial-policy`
- `/methodology`
- `/data-sources`
- `/disclaimer`

### 3단계: 핵심 콘텐츠 작성

- 사례형 7개 작성
- 가이드형 8개 작성
- 모든 글에 작성일, 업데이트일, 계산 기준일, 출처, 면책 고지, 관련 계산기 링크를 포함한다.

### 4단계: 계산기 신뢰도 보강

- 결과 화면에 반영된 항목과 반영되지 않은 항목을 표시한다.
- 계산 기준과 출처 링크를 연결한다.
- 계산 결과가 참고용임을 명확히 고지한다.

### 5단계: 기술 SEO 정리

- sitemap, robots, canonical, metadata를 정리한다.
- Article, FAQPage, WebApplication JSON-LD를 추가한다.
- footer 정책 링크와 내부 링크 구조를 정리한다.

### 6단계: 색인 및 품질 확인

- Search Console에 등록한다.
- 주요 URL 색인 요청을 한다.
- 모바일 표시, 깨진 링크, 빈 페이지, 중복 metadata를 점검한다.

### 7단계: AdSense 재심사 준비

- 재심사 직전에 AdSense 심사용 코드를 삽입한다.
- 승인 전 광고 슬롯 UI는 만들지 않는다.
- 모든 주요 페이지가 접근 가능하고 색인 가능한 상태인지 확인한다.

## 재심사 전 체크리스트

- [ ] 홈에서 계산기 외 콘텐츠 섹션이 충분히 보인다.
- [ ] 신뢰 페이지 8개가 모두 실제 본문을 갖췄다.
- [ ] 사례 7개와 가이드 8개가 독립 URL로 존재한다.
- [ ] 각 콘텐츠에 작성일, 업데이트일, 계산 기준일, 출처, 면책 고지가 있다.
- [ ] `/data-sources`가 항목별 표로 정리되어 있다.
- [ ] 계산 결과에 반영 항목과 미반영 항목이 표시된다.
- [ ] 개인정보처리방침에 쿠키, 제3자 광고, Google 관련 고지가 있다.
- [ ] sitemap과 robots가 정상 동작한다.
- [ ] 모든 주요 페이지의 metadata와 canonical이 있다.
- [ ] 모바일에서 본문, footer, 정책 링크가 깨지지 않는다.
- [ ] 빈 페이지, 공사 중 문구, placeholder 텍스트가 없다.
- [ ] Search Console에서 주요 URL이 색인 가능 상태다.
- [ ] AdSense 코드는 재심사 직전에 최소 형태로 들어간다.

## 승인 가능성을 낮추는 금지사항

- 계산기만 있고 읽을 만한 본문이 부족한 구조
- 계산 결과를 그대로 복사한 얇은 사례 글
- 출처 없는 세금, 대출, 보증보험 설명
- 투자 수익 보장처럼 보이는 표현
- "무조건 매수/전세/월세가 유리하다"는 단정
- 작성자, 연락처, 개인정보처리방침이 불명확한 상태
- 빈 카테고리, 준비 중 페이지, 깨진 링크
- 광고 슬롯이 본문보다 먼저 보이는 배치
- 계산 전 화면, 로딩 화면, 버튼 주변 광고
- 동일한 문장 구조로 양산한 콘텐츠
- AI 생성 티가 강한 일반론 중심 글
- 모바일에서 본문보다 CTA나 광고가 과도하게 보이는 구성

## 3회 피드백 루프 요약

### 1차 피드백 반영

초기안은 콘텐츠 수와 페이지 추가에 치우쳐 있었다. 1차 피드백에서 홈, 가이드 허브, footer, 정책 페이지가 하나의 완성된 서비스처럼 보여야 한다는 점을 반영했다. 또한 YMYL 성격에 맞춰 편집 정책, 업데이트 일자, 출처, 면책 고지를 페이지 단위로 설계에 추가했다.

### 2차 피드백 반영

사례 콘텐츠가 계산기 결과 반복으로 보일 위험이 지적됐다. 이에 따라 사례 주제를 단순 가격대가 아니라 전세가율, 대출금리, 월세 부담, 보유 기간, 초기자금 같은 의사결정 상황 중심으로 재설계했다. 또한 `/data-sources`를 항목별 출처 관리 표로 구체화했다.

### 3차 피드백 반영

최종적으로 페이지 수보다 심사 가능한 완성도를 기준으로 문서화했다. 콘텐츠 작성, 신뢰 페이지 완성, 기술 SEO, 색인 확인, AdSense 코드 삽입, 재심사 요청 순서로 실행 로드맵을 확정했다.
